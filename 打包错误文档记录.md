# Debian 打包错误文档记录

本文档记录 Launcher 项目 Debian 打包过程中遇到的问题及解决方案。

---

## 错误 1：debian/control 格式错误

### 错误信息
```
dpkg-buildpackage: error: syntax error in debian/control at line 16: block lacks the 'Package' field
```

### 错误原因
`debian/control` 文件中，源包信息块（Source）与二进制包信息块（Package）之间的空行位置不正确。Build-Depends 字段后多了一个空行，导致解析器在第 16 行找不到 `Package` 字段。

### 文件格式要求
`debian/control` 文件由多个段落（块）组成，每个块之间必须用**恰好一个空行**分隔：
1. **源包块**：以 `Source:` 开头，包含源包的元信息
2. **二进制包块**：以 `Package:` 开头，包含二进制包的元信息

### 错误示例
```
Source: launcher
...
Build-Depends: debhelper-compat (= 13),
                 cmake
                  <-- 多余的空行
Standards-Version: 4.6.0
...
                  <-- 正确的分隔空行
Package: launcher
```

### 解决方案
确保源包信息块与二进制包信息块之间只有一个空行，删除多余空行。

## 错误 2：debhelper compat 级别重复定义

### 错误信息
```
dh: error: debhelper compat level specified both in debian/compat and via build-dependency on debhelper-compat
```

### 错误原因
debhelper 兼容级别被指定了两次：
1. `debian/compat` 文件中指定了 compat 级别（内容为 `13`）
2. `debian/control` 的 `Build-Depends` 中通过 `debhelper-compat (= 13)` 指定了 compat 级别

### 背景知识
debhelper 兼容级别有两种指定方式（只能选其一）：
- **旧方式**：创建 `debian/compat` 文件，内容为兼容级别数字
- **新方式（推荐）**：在 `debian/control` 的 `Build-Depends` 中添加 `debhelper-compat (= N)`

### 解决方案
使用新方式，删除 `debian/compat` 文件，只保留 `debian/control` 中的 `debhelper-compat (= 13)` 依赖。

```bash
rm debian/compat
```

## 错误 3：修复未生效（网络问题）

### 现象
本地已修复并提交，但 GitHub Actions 构建仍然报错：
```
dh: error: debhelper compat level specified both in debian/compat and via build-dependency on debhelper-compat
```

### 错误原因
由于网络问题，本地提交未能成功推送到远程仓库。`git push` 命令超时失败，但错误信息可能被忽略。

### 排查方法
1. 检查远程仓库状态：`git log origin/main --oneline -3`
2. 对比本地与远程：`git log main..origin/main`
3. 确认推送结果：推送成功后会显示 `xxx..yyy main -> main`

### 解决方案
重新执行 `git push origin main`，确保看到类似输出：
```
To https://github.com/xxx/xxx.git
   23c382b..72be7b9  main -> main
```

### 经验教训
- 修复后必须确认推送成功，不要只看本地提交
- GitHub Actions 会运行远程仓库的代码，而非本地代码

## 错误 4：Workflow 使用旧 commit 构建

### 现象
本地已修复并成功推送到远程，但 GitHub Actions 仍然使用旧 commit 构建，报错：
```
dh: error: debhelper compat level specified both in debian/compat and via build-dependency on debhelper-compat
```

通过 API 检查远程仓库，`debian/compat` 文件确实已删除。

### 错误原因
Workflow 配置只在以下情况触发：
```yaml
on:
  push:
    tags:
      - 'v*'
  workflow_dispatch:
```

这意味着：
1. **push 到 main 分支不会自动触发构建**
2. 只有打 tag 或手动触发才会运行
3. 之前的 workflow 运行使用的是旧 commit (23c382b)，修复 commit (72be7b9) 从未被构建

### 排查方法
```bash
# 检查远程仓库文件
curl -s "https://api.github.com/repos/OWNER/REPO/contents/debian" | jq '.[].name'

# 检查 workflow 运行使用的 commit
curl -s "https://api.github.com/repos/OWNER/REPO/actions/runs?per_page=3" | \
  jq '.workflow_runs[] | {id, head_sha, status, conclusion}'
```

### 解决方案
修改 `.github/workflows/build.yml`，添加 main 分支 push 触发：
```yaml
on:
  push:
    branches:
      - main
    tags:
      - 'v*'
  workflow_dispatch:
```

或者手动触发 workflow：
1. 打开 GitHub Actions 页面
2. 选择 "Build Debian Package" workflow
3. 点击 "Run workflow"

### 经验教训
- 检查 workflow 的触发条件，确保修复后能触发新构建
- 使用 GitHub API 验证远程仓库的实际状态
- 区分"本地代码"、"远程代码"、"workflow 实际运行的代码"三者

## 错误 5：CMake install 找不到资源文件

### 错误信息
```
CMake Error at cmake_install.cmake:62 (file):
  file INSTALL cannot find
  "/home/runner/work/launcher-uos/launcher-uos/resources/data/launcher.desktop":
  No such file or directory.
```

### 错误原因
CMakeLists.txt 中配置了安装资源文件：
```cmake
install(FILES resources/data/launcher.desktop
    DESTINATION share/applications
)

install(FILES resources/icons/launcher.svg
    DESTINATION share/icons/hicolor/scalable/apps
)
```

但实际的 `resources/data/` 和 `resources/icons/` 目录中缺少这些文件：
- `resources/data/launcher.desktop` - 不存在
- `resources/icons/launcher.svg` - 不存在

### 排查方法
```bash
# 检查 CMakeLists.txt 中的 install 指令
grep -n "install(FILES" CMakeLists.txt

# 检查资源文件是否存在
ls -la resources/data/
ls -la resources/icons/
```

### 解决方案
创建缺失的资源文件：

1. **launcher.desktop** - Desktop Entry 文件：
```ini
[Desktop Entry]
Version=1.0
Name=Launcher
Name[zh_CN]=聚焦搜索
Exec=Launcher
Icon=launcher
Terminal=false
Type=Application
Categories=Utility;System;Qt;
```

2. **launcher.svg** - 应用图标（SVG 格式）

### 经验教训
- 在添加 `install(FILES ...)` 指令前，确保文件已存在
- 本地构建可能跳过某些 install 步骤，CI 环境会完整执行
- 使用 `git ls-files` 检查哪些文件被跟踪

## 最佳实践总结

### debian/control 文件规范
1. 源包块以 `Source:` 开头
2. 每个二进制包块以 `Package:` 开头
3. 块之间用单个空行分隔
4. 字段续行使用空格缩进（不能是 Tab）

### debhelper 兼容级别
1. 优先使用 `debian/control` 中的 `debhelper-compat (= N)` 方式
2. 不要创建 `debian/compat` 文件
3. 当前推荐使用 compat 级别 13（对应 debhelper 13.x）

### 相关文件检查清单
- [ ] `debian/control` - 包元信息，格式正确，无多余空行
- [ ] `debian/rules` - 构建规则，可执行权限
- [ ] `debian/changelog` - 变更日志，格式正确
- [ ] `debian/postinst` - 安装后脚本（可选）
- [ ] `debian/prerm` - 卸载前脚本（可选）
- [ ] 不要有 `debian/compat` 文件（使用 control 中的声明方式）
